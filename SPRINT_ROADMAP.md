# Sprint Roadmap

This roadmap continues the existing plan for the offline-first, single-branch, cash-only fast-food POS.
It assumes the current repository is at the bootstrap stage: monorepo structure, basic shared DTOs,
minimal fiscal/cash/sync helpers, worker stubs, and smoke tests are present.

## Delivery rhythm

- Sprint length: 2 weeks.
- Release target: usable single-branch cash POS after Sprint 6, hardened beta after Sprint 8.
- Scope guardrails:
  - no multi-branch features;
  - no SATIM/CIB/Edahabia payment integration;
  - cash only;
  - offline-first local sale writes;
  - VAT-disabled fiscal engine v1;
  - no direct printer calls from the POS client.

## Sprint 1: Domain Completion foundation

### Goal

Turn `packages/shared-types` from smoke-test DTOs into the stable contract layer for the first cash-sale
vertical slice.

### Main work

- Add core primitives and status unions:
  - `OrderId`, `ProductId`, `PaymentId`, `ReceiptId`, `PrinterJobId`, `SyncEventId`;
  - `IsoDateTimeString`, `IdempotencyKey`, `FiscalVersion`, `CurrencyDZD`;
  - `OrderStatus`, `PaymentStatus`, `PrinterJobStatus`, `SyncEventType`.
- Add `Product` and `ProductCategory` DTOs.
- Expand `OrderItem` to include immutable product snapshots.
- Expand `Order`, `CashPayment`, `Receipt`, `SyncEvent`, and `PrinterJob` enough to represent one complete cash sale.
- Update smoke tests to build one complete in-memory sale fixture.

### Acceptance criteria

- A full finalized local cash sale can be represented only with shared DTOs.
- No multi-branch or non-cash payment fields are introduced.
- `npm run check` passes.

## Sprint 2: Fiscal Engine v1 stabilization

### Goal

Make the VAT-disabled fiscal engine deterministic, validated, line-aware, and regression-testable.

### Main work

- Replace subtotal-only `calculateReceipt(orderId, subtotalDZD)` with structured `calculateReceipt(input)`.
- Add fiscal input/output DTOs from Sprint 1 contracts.
- Implement line-level calculation:
  - line subtotal;
  - line VAT at 0%;
  - line total;
  - receipt subtotal/VAT/total.
- Add a single rounding helper and document the v1 rounding policy.
- Add explicit validation errors for invalid fiscal input.
- Add regression fixtures for one-line, multi-line, and decimal-price receipts.

### Acceptance criteria

- Receipt line totals reconcile exactly with receipt totals.
- Invalid inputs fail before receipt generation.
- Fiscal version is always `v1`.
- Regression tests protect expected v1 outputs.
- `npm run check` passes.

## Sprint 3: Local offline persistence and outbox contracts

### Goal

Prepare the POS desktop app to write every sale locally before any server sync attempt.

### Main work

- Add a local persistence boundary in `apps/pos-desktop`:
  - `LocalOrderRepository` interface;
  - `LocalPaymentRepository` interface;
  - `LocalReceiptRepository` interface;
  - `LocalOutboxRepository` implementation contract.
- Define SQLite table shape documentation for:
  - `orders`;
  - `order_items`;
  - `cash_payments`;
  - `receipts`;
  - `sync_outbox`.
- Add an in-memory repository for tests before binding to Tauri SQLite.
- Implement `finalizeCashSale` service that writes order, receipt, payment, and outbox events transactionally.

### Acceptance criteria

- Finalized sale path does not require API/network availability.
- Tests prove sale data and outbox events are created together.
- Failed sync does not lose local sale data.
- `npm run check` passes.

## Sprint 4: API sync ingestion and idempotency

### Goal

Create the server-side sync boundary for accepting offline-generated sale events safely.

### Main work

- Expand `apps/api` sync module around typed event ingestion.
- Add DTO validation at API boundary.
- Implement idempotency behavior by `idempotencyKey`.
- Add server-side persistence interfaces for synced orders, payments, receipts, and print-job requests.
- Keep implementation storage simple/in-memory if database setup is not ready, but design interfaces for PostgreSQL later.

### Acceptance criteria

- Duplicate sync events are safely rejected or acknowledged without double effects.
- Events are processed in a deterministic order.
- API accepts sale events generated by Sprint 3.
- `npm run check` passes.

## Sprint 5: Server-side print queue boundary

### Goal

Keep all printing server/worker-side and prepare receipt print jobs for later ESC/POS implementation.

### Main work

- Expand `Printer` and `PrinterJob` contracts.
- Add print job creation after receipt sync or sale finalization policy is chosen.
- Implement worker-side print job processor states:
  - queued;
  - processing;
  - sent;
  - failed;
  - dead-lettered.
- Add retry metadata and failure reason fields.
- Keep ESC/POS calls stubbed behind a printer transport interface.

### Acceptance criteria

- POS client never sends TCP/ESC/POS commands directly.
- Worker can process a queued print job through a test transport.
- Failed jobs preserve retry/failure metadata.
- `npm run check` passes.

## Sprint 6: POS cash register vertical slice

### Goal

Build the first usable desktop cash register flow on top of the stabilized domain, fiscal, local persistence,
sync, and print boundaries.

### Main work

- Add initial React/Tauri app shell if not yet present.
- Implement product list and cart state.
- Implement cash payment screen:
  - amount due;
  - received amount;
  - change due;
  - underpayment guard.
- Connect finalization to local offline sale service.
- Show sale state:
  - saved locally;
  - pending sync;
  - synced;
  - sync failed.

### Acceptance criteria

- Cashier can create and finalize a cash sale offline.
- Receipt totals come from Fiscal Engine v1.
- Sale appears in local pending-sync state after finalization.
- No payment integration or multi-branch UI appears.
- `npm run check` passes.

## Sprint 7: Sync worker and reconnect behavior

### Goal

Make pending local sales sync automatically and safely when the API becomes reachable.

### Main work

- Implement POS-side outbox flush loop:
  - startup replay;
  - batch size;
  - retry backoff;
  - failure state;
  - sync acknowledgments.
- Add network/API availability abstraction.
- Add tests for partial sync failure and retry.
- Add simple sync status UI hooks.

### Acceptance criteria

- Pending events survive app restart.
- Duplicate sends do not create duplicate server records.
- Partial failures retry without marking unsynced events as synced.
- `npm run check` passes.

## Sprint 8: Beta hardening and operator readiness

### Goal

Make the single-branch, cash-only POS ready for controlled beta use.

### Main work

- Add structured logs for sale finalization, sync, fiscal calculation, and print jobs.
- Add backup/restore notes for local SQLite data.
- Add basic seed data for categories/products.
- Add operator-facing failure states:
  - printer unavailable;
  - sync backlog;
  - local persistence error.
- Add end-to-end smoke checklist for a no-network sale and later sync.

### Acceptance criteria

- A test operator can run a full offline sale and later sync it.
- Basic recovery instructions exist for local data and print failures.
- No direct client printer calls exist.
- `npm run check` passes.

## Recommended immediate next action

Start Sprint 1 by editing `packages/shared-types/src/index.ts` and `tests/run-tests.ts` to implement the
complete shared contracts for one finalized cash sale. Do not start UI, SQLite, or API persistence until the
shared DTOs and fiscal receipt contracts are stable enough to avoid rework.
